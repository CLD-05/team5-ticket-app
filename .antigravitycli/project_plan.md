## 03. 핵심 기능 / 부가 기능

### 핵심 기능 (필수)

1. **가상 대기열 토큰 발급 및 검증** - 트래픽 진입 통제 및 순차 진입을 위한 Redis 기반 대기열 시스템 구현
2. **좌석 임시 선점 (Hold)** - 동시성 제어를 위한 Redis 캐시 기반의 5분 임시 선점 처리 및 동기 응답 제공
3. **비동기 예매 처리 (Booking)** - AWS SQS 메시지 큐를 통해 대량 쓰기 요청을 완충하여 비동기로 데이터베이스 저장
4. **공연 및 좌석 실시간 조회** - 임시 선점 상태가 실시간으로 반영된 예약 가능 좌석 조회 기능

### 부가 기능 (여유 시)

1. **실시간 대기 순번 알림** - 사용자가 대기 페이지에서 실시간으로 대기 순위 및 예상 대기 시간을 조회하는 기능
2. **어드민 콘솔** - 신규 공연 등록, 좌석 일괄 생성 및 대기열 동적 임계값 관리 기능

## 04. 아키텍처 개요

```
[ 사용자 ] -> [ 프론트엔드 (Thymeleaf/HTML) ] -> [ 백엔드 API (Spring Boot) ]
                                                    |
                                                    +---> [ Redis (대기열 & 임시선점) ]
                                                    |
                                                    +---> [ AWS SQS (예매 요청 큐) ] -> [ Booking Worker (백엔드) ] -> [ MySQL (AWS RDS) ]
```

**구성 요소:**
* **프론트** : Thymeleaf, Vanilla HTML/CSS/JS
* **백엔드** : Spring Boot 3.2.3, Spring Security (JWT)
* **데이터** : MySQL (AWS RDS), Redis (ElastiCache)
* **외부연동 / 큐** : AWS SQS (Simple Queue Service)

## 06. 환경·인프라 설계 결정

```
dev/prod 분리   : dev와 prod용 VPC 및 Kubernetes 네임스페이스를 분리하여 환경 격리 운영
RDS 구성        : dev는 단일 DB 사용, prod는 고가용성을 위해 Multi-AZ를 사용하고 읽기 부하 분산을 위한 Read Replica 구성
앱 개수/라우팅  : 클라이언트 요청을 받는 Web API App과 SQS 메시지를 소비하는 Booking Worker App으로 다중화하여 독립 스케일링 구성
관측성 범위     : 응답 지연 시간, TPS, 에러율, CPU/메모리 사용률 및 SQS 대기 메시지 수 임계치 모니터링
오토스케일      : KEDA를 연동하여 SQS 큐에 쌓인 예약 대기 메시지 수에 비례하여 Booking Worker Pod를 동적 오토스케일링
```

## 10. 리스크 / 미정 사항

* **로컬 개발 SQS 테스트 환경:** 팀원들이 각자의 로컬 PC에서 AWS 연결 없이 워커 동작을 테스트할 수 있도록 LocalStack 도커 컨테이너 구성 및 환경 스위칭 방안 조율 필요
* **Redis 메모리 관리:** 대규모 트래픽 진입 시 발급된 가상 대기열 토큰의 폭증으로 인한 Redis 메모리 풀 방지용 토큰 자동 만료/정리 로직 최적화 필요
* **AWS 프리티어 및 배포 인스턴스 사양:** 부하 테스트 및 모니터링 모듈이 안정적으로 구동될 수 있는 인프라 최소 사양 정의 및 예산 범위 검토 필요
